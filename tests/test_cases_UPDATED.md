# Тест-кейсы WorkTime Sync

## TC-01. Healthcheck backend

**Цель:** проверить, что backend запущен.

**Шаги:**

1. Запустить backend.
2. Открыть `GET /health`.

**Ожидаемый результат:**

```json
{"status": "ok"}
```

---

## TC-02. Главный endpoint для frontend

**Цель:** проверить, что frontend может получить все данные одним запросом.

**Шаги:**

1. Выполнить `GET /api/worktime/overview`.
2. Проверить структуру ответа.

**Ожидаемый результат:**

Ответ содержит:

```text
employees
events
roadmap
summary
recommendations
bestSlots
notifications
groups
```

---

## TC-03. Загрузка данных из data/synthetic

**Цель:** проверить, что backend использует основной источник данных `data/synthetic`.

**Шаги:**

1. Открыть `data/synthetic/employees.csv`.
2. Проверить, что там 10 сотрудников.
3. Выполнить `GET /employees`.
4. Сравнить количество сотрудников в API с CSV.

**Ожидаемый результат:**

- API возвращает сотрудников из `data/synthetic/employees.csv`;
- количество сотрудников совпадает с активным CSV;
- старый `backend/data` не является основным источником.

---

## TC-04. Динамическое количество сотрудников и команд

**Цель:** проверить, что сотрудники и отделы не заданы вручную в коде.

**Шаги:**

1. Выполнить `GET /analytics/summary`.
2. Проверить количество сотрудников.
3. Выполнить `GET /analytics/groups`.
4. Проверить, что группы строятся на основе данных.
5. Проверить команды из поля `team`.

**Ожидаемый результат:**

- количество сотрудников считается из данных;
- команды берутся из поля `team`;
- при изменении CSV следующий запрос пересчитывает аналитику.

---

## TC-05. Сотрудник с устаревшим графиком

**Цель:** проверить расчёт актуальности графика.

**Шаги:**

1. Выполнить `GET /employees`.
2. Найти сотрудника с давней датой `last_update_date`.
3. Проверить `days_since_update`.
4. Проверить `schedule_actuality`.
5. Проверить `risk`.

**Ожидаемый результат:**

- чем старее дата обновления, тем ниже актуальность;
- риск сотрудника повышается;
- сотрудник может попасть в группу `outdated` или `needsConfirmation`.

---

## TC-06. Встреча вне рабочего времени

**Цель:** проверить поиск календарных конфликтов.

**Шаги:**

1. Выполнить `GET /analytics/conflicts`.
2. Проверить список конфликтов.
3. Сравнить время события с рабочим графиком сотрудника.

**Ожидаемый результат:**

- endpoint возвращает события вне рабочего графика;
- для каждого конфликта есть сотрудник, событие, день, время, причина и severity.

---

## TC-07. Перегруженный сотрудник

**Цель:** проверить расчёт загрузки.

**Шаги:**

1. Выполнить `GET /employees`.
2. Найти сотрудника с высокой `load`.
3. Проверить `busy_hours`, `work_hours`, `load`.
4. Проверить рекомендации и группы.

**Ожидаемый результат:**

- `load = busy_hours / work_hours`;
- сотрудник с высокой загрузкой попадает в `highLoad`;
- появляются рекомендации или уведомления.

---

## TC-08. HR/data mismatch

**Цель:** проверить расхождения между рабочим графиком и HR-профилем.

**Шаги:**

1. Выполнить `GET /analytics/data-mismatches`.
2. Проверить список расхождений.
3. Выполнить `GET /analytics/groups`.
4. Проверить группы `timezoneConflict` и `hrMismatch`.

**Ожидаемый результат:**

- endpoint возвращает расхождения;
- сотрудник с расхождением попадает в нужную группу;
- риск учитывает `timezone_mismatch` и `hr_mismatch`.

---

## TC-09. Карта доступности

**Цель:** проверить командную доступность.

**Шаги:**

1. Выполнить `GET /analytics/availability`.
2. Проверить дни и слоты.
3. Проверить `missing` и `missingDetails`.

**Ожидаемый результат:**

- карта содержит рабочие дни и часы;
- каждый слот показывает количество доступных сотрудников;
- причины недоступности отображаются в `missingDetails`.

---

## TC-10. Лучшие окна встречи

**Цель:** проверить подбор meeting slots.

**Шаги:**

1. Выполнить `GET /meeting-slots`.
2. Проверить количество слотов.
3. Проверить поля слота.

**Ожидаемый результат:**

Ответ содержит от 1 до 3 лучших слотов.

Поля:

```text
label
count
missing
missingDetails
```

## TC-14. Загрузка CSV/JSON

**Цель:** проверить загрузку нового dataset.

**Шаги:**

1. Подготовить корректный CSV или JSON.
2. Выполнить `POST /upload/employees` или другой dataset.
3. Выполнить `GET /employees`.

**Ожидаемый результат:**

- файл сохраняется в `data/synthetic`;
- данные валидируются;
- следующий запрос к API использует новый файл;
- при загрузке CSV он имеет приоритет над JSON.

---

## TC-15. Некорректная загрузка

**Цель:** проверить Pydantic-валидацию.

**Шаги:**

1. Подготовить CSV без обязательного поля, например `id` или `work_start`.
2. Выполнить `POST /upload/employees`.

**Ожидаемый результат:**

- backend отклоняет файл;
- возвращается ошибка;
- невалидный файл не становится активным источником данных.

---
