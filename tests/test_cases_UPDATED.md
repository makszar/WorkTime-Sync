# Тест-кейсы WorkTime Sync

## TC-01. Проверка healthcheck

**Цель:** убедиться, что backend запущен.

**Шаги:**

1. Запустить backend.
2. Открыть `GET /health`.

**Ожидаемый результат:**

```json
{"status": "ok"}
```

---

## TC-02. Проверка главного endpoint для frontend

**Цель:** проверить, что frontend может получить все данные одним запросом.

**Шаги:**

1. Выполнить `GET /api/worktime/overview`.
2. Проверить структуру ответа.

**Ожидаемый результат:**

Ответ содержит:

```text
employees
events
roadmap
summary
recommendations
bestSlots
availability
dataMismatches
```

---

## TC-03. Сотрудник с устаревшим графиком

**Цель:** проверить, что система определяет сотрудника, который давно не обновлял график.

**Шаги:**

1. Выполнить `GET /employees`.
2. Найти сотрудника с большой разницей между `DEMO_TODAY` и `last_update_date`.
3. Проверить `metrics.days_since_update`.
4. Проверить `metrics.schedule_actuality`.
5. Проверить `metrics.risk`.
6. Проверить `metrics.risk_status`.

**Ожидаемый результат:**

- `days_since_update` высокий;
- `schedule_actuality` ниже, чем у недавно обновлённых сотрудников;
- риск растёт;
- сотрудник попадает в группу среднего/высокого/критического риска.

---

## TC-04. Встреча вне рабочего времени

**Цель:** проверить поиск календарного конфликта.

**Шаги:**

1. Выполнить `GET /analytics/conflicts`.
2. Проверить, что ответ — список конфликтов.
3. Проверить поля первого конфликта.

**Ожидаемый результат:**

Конфликт содержит:

```text
employee
title
day
time
reason
severity
```

Событие должно быть связано со встречей вне рабочего времени.

---

## TC-05. Перегруженный сотрудник

**Цель:** проверить расчёт загрузки.

**Шаги:**

1. Выполнить `GET /employees`.
2. Выбрать сотрудника с большим количеством занятых часов.
3. Проверить `metrics.busy_hours`.
4. Проверить `metrics.work_hours`.
5. Проверить `metrics.load`.

**Ожидаемый результат:**

- `load = busy_hours / work_hours`;
- если `load >= 0.8`, сотрудник считается близким к перегрузке;
- риск сотрудника повышается;
- в рекомендациях может появиться предупреждение о нагрузке.

---

## TC-06. Подбор лучшего окна встречи

**Цель:** проверить, что система предлагает лучшие командные слоты.

**Шаги:**

1. Выполнить `GET /meeting-slots`.
2. Проверить количество слотов.
3. Проверить структуру первого слота.

**Ожидаемый результат:**

Ответ содержит от 1 до 3 слотов.

Каждый слот содержит:

```text
label
count
missing
missingDetails
```

Слоты сортируются по количеству доступных сотрудников и учитывают рабочие часы, события, отсутствия и перегруз.

---

## TC-07. Командная доступность

**Цель:** проверить карту доступности.

**Шаги:**

1. Выполнить `GET /analytics/availability`.
2. Проверить, что возвращаются рабочие дни.
3. Проверить наличие `slots`.
4. Проверить структуру одного слота.

**Ожидаемый результат:**

Каждый слот содержит:

```text
hour
count
type
missing
missingDetails
```

Если сотрудник недоступен, в `missingDetails` указана причина.

---

## TC-08. Рекомендация для HR

**Цель:** проверить рекомендации при расхождении HR-данных и рабочего графика.

**Шаги:**

1. Выполнить `GET /analytics/data-mismatches`.
2. Проверить список расхождений.
3. Выполнить `GET /recommendations`.
4. Найти рекомендацию по HR/часовому поясу.

**Ожидаемый результат:**

- система показывает расхождение часового пояса или HR-профиля;
- риск сотрудника повышается;
- рекомендация объясняет, что нужно сверить HR-данные.

---

## TC-10. Несуществующий сотрудник

**Цель:** проверить обработку ошибки.

**Шаги:**

1. Выполнить `GET /employees/999`.

**Ожидаемый результат:**

- HTTP 404;
- ошибка: `Сотрудник не найден`.

---

## TC-11. Загрузка JSON/CSV

**Цель:** проверить загрузку нового набора данных.

**Шаги:**

1. Подготовить корректный `employees.csv` или `employees.json`.
2. Выполнить `POST /upload/employees`.
3. Выполнить `GET /employees`.

**Ожидаемый результат:**

- backend сохраняет файл;
- данные проходят Pydantic-валидацию;
- новый набор сотрудников используется в API;
- если загружен CSV, он используется вместо JSON.

---

## TC-12. Динамическое количество сотрудников и команд

**Цель:** проверить, что количество сотрудников и отделов берётся из данных.

**Шаги:**

1. Добавить нового сотрудника в `employees.json` или загрузить новый `employees.csv`.
2. Выполнить `GET /analytics/summary`.
3. Проверить `total_employees`.
4. Проверить список `teams`.

**Ожидаемый результат:**

- `total_employees` увеличивается;
- новая команда появляется в `teams`, если в данных есть новое значение поля `team`;
- код не требует ручного изменения количества сотрудников.

---

## TC-13. Backend автотесты

**Цель:** проверить существующий набор backend-тестов.

**Шаги:**

1. Перейти в папку backend.
2. Выполнить:

```bash
pytest
```

**Ожидаемый результат:**

Тесты проходят успешно и проверяют:

- healthcheck;
- контракт overview;
- конфликты;
- доступность;
- meeting slots;
- data mismatches.
