# Тест-кейсы WorkTime Sync

## TC-01. Сотрудник с устаревшим графиком

**Цель:** проверить, что система определяет сотрудника, который давно не обновлял рабочий график.

**Предусловия:**

- backend запущен;
- в `employees.json` есть сотрудник с `last_update_date`, который даёт большое значение `days_since_update`.

**Шаги:**

1. Открыть backend Swagger: `http://127.0.0.1:8000/docs`.
2. Выполнить запрос `GET /employees`.
3. Найти сотрудника с давней датой `last_update_date`.
4. Проверить поле `metrics.days_since_update`.
5. Проверить поле `metrics.schedule_actuality`.
6. Проверить поле `metrics.risk`.

**Ожидаемый результат:**

- `days_since_update` больше 60 или 90 дней;
- `schedule_actuality` снижается;
- `risk` растёт;
- `risk_status` становится средним, высоким или критическим;
- в карточке сотрудника появляется рекомендация обновить рабочий график.

---

## TC-02. Встреча вне рабочего времени

**Цель:** проверить, что система находит события, которые выходят за рамки рабочего графика.

**Предусловия:**

- backend запущен;
- в `events.json` есть событие, которое начинается раньше `work_start` или заканчивается позже `work_end`.

**Шаги:**

1. Выполнить запрос `GET /employees/{employee_id}` для сотрудника с конфликтным событием.
2. Посмотреть список `events`.
3. Сравнить время события с `work_start` и `work_end`.
4. Проверить `metrics.outside_work_events`.
5. Проверить `metrics.outside_work_ratio`.
6. Проверить `metrics.conflict_count`.

**Ожидаемый результат:**

- система считает событие конфликтным;
- `outside_work_events` больше 0;
- `outside_work_ratio` больше 0;
- `conflict_count` увеличивается;
- рекомендация предлагает проверить или перенести встречи вне рабочего времени.

---

## TC-03. Перегруженный сотрудник

**Цель:** проверить, что система определяет сотрудника с высокой загрузкой.

**Предусловия:**

- у сотрудника есть календарные события с большой суммарной длительностью;
- `busy_hours / work_hours` приближается к 0.8 или выше.

**Шаги:**

1. Выполнить `GET /employees`.
2. Выбрать сотрудника с большим количеством событий или длинными focus-блоками.
3. Проверить `metrics.busy_hours`.
4. Проверить `metrics.work_hours`.
5. Проверить `metrics.load`.
6. Выполнить `GET /employees/{employee_id}`.
7. Проверить список рекомендаций.

**Ожидаемый результат:**

- `load` рассчитан по формуле `busy_hours / work_hours`;
- если `load >= 0.8`, сотрудник считается близким к перегрузке;
- система добавляет рекомендацию проверить загрузку;
- риск сотрудника увеличивается.

---

## TC-04. Подбор лучшего окна встречи

**Цель:** проверить сценарий подбора оптимального окна для командной встречи.

**Текущее состояние реализации:**

- во frontend есть расчёт лучших окон встречи на моковых данных;
- в backend отдельный endpoint `/meeting-slots` пока не реализован.

**Шаги для frontend:**

1. Запустить frontend.
2. Открыть страницу командной доступности.
3. Проверить, что система показывает лучшие окна для встречи.
4. Проверить, что учитываются рабочие дни, часы и загрузка сотрудников.
5. Сравнить отображаемые окна с данными сотрудников.

**Ожидаемый результат:**

- frontend показывает 2–3 лучших окна;
- окна выбираются на основе количества доступных сотрудников;
- если сотрудник недоступен, он попадает в список отсутствующих.

**Ожидаемый результат для backend после доработки:**

- endpoint `GET /meeting-slots` должен возвращать 2–3 лучших слота;
- слоты должны учитывать рабочие часы, часовые пояса, отсутствия, события и загрузку.

## TC-05. Рекомендация для HR

**Цель:** проверить, что система формирует HR-рекомендацию при расхождении данных.

**Предусловия:**

- в `hr_profiles.json` есть HR-профиль сотрудника;
- данные HR-профиля отличаются от рабочего графика сотрудника по времени, формату или часовому поясу.

**Шаги:**

1. Выполнить запрос `GET /employees/{employee_id}`.
2. Сравнить данные `employee` и `hr_profile`.
3. Проверить `metrics.timezone_mismatch`.
4. Проверить `metrics.hr_mismatch`.
5. Проверить список `recommendations`.

**Ожидаемый результат:**

- если часовой пояс отличается, `timezone_mismatch = 1`;
- если формат или часы отличаются, `hr_mismatch = 1`;
- риск сотрудника увеличивается;
- система предлагает сверить часовой пояс или согласовать данные с HR.

---

## TC-06. Общая аналитика команды

**Цель:** проверить, что backend корректно собирает summary по команде.

**Шаги:**

1. Выполнить `GET /analytics/summary`.
2. Проверить `total_employees`.
3. Проверить `avg_risk`.
4. Проверить `avg_schedule_actuality`.
5. Проверить `avg_load`.
6. Проверить `total_conflicts`.
7. Проверить `risk_distribution`.
8. Проверить аналитику по командам.

**Ожидаемый результат:**

- summary содержит агрегированные показатели;
- количество сотрудников совпадает с `employees.json`;
- распределение риска соответствует метрикам сотрудников;
- total_conflicts учитывает конфликты рабочего времени, часового пояса и HR-данных.

## TC-07. Проверка API карточки сотрудника

**Цель:** проверить, что карточка сотрудника возвращает полный набор данных.

**Шаги:**

1. Выполнить `GET /employees/1`.
2. Проверить наличие блока `employee`.
3. Проверить наличие блока `hr_profile`.
4. Проверить наличие блока `events`.
5. Проверить наличие блока `metrics`.
6. Проверить наличие блока `recommendations`.

**Ожидаемый результат:**

Ответ содержит все ключевые блоки, необходимые для отображения карточки сотрудника и объяснения риска.

## TC-08. Сотрудник не найден

**Цель:** проверить обработку ошибки при запросе несуществующего сотрудника.

**Шаги:**

1. Выполнить `GET /employees/999`.
2. Посмотреть статус ответа.

**Ожидаемый результат:**

- backend возвращает HTTP 404;
- текст ошибки: `Сотрудник не найден`.

## TC-09. Проверка загрузки данных из JSON

**Цель:** проверить, что backend читает данные из JSON-файлов.

**Шаги:**

1. Открыть `backend/data/employees.json`.
2. Добавить тестового сотрудника.
3. Перезапустить backend или повторить запрос, если данные перечитываются при каждом запросе.
4. Выполнить `GET /employees`.

**Ожидаемый результат:**

- новый сотрудник появляется в ответе API;
- для него рассчитываются метрики;
- если данных событий нет, конфликтные показатели равны 0.

**Цель:** зафиксировать проверку будущей возможности загрузки CSV.

**Ожидаемый результат после доработки:**

- backend валидирует структуру;
