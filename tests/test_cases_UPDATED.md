# Тест-кейсы WorkTime Sync

Файл содержит ручные тест-кейсы для проверки MVP. Их можно проходить через frontend, Swagger (`/docs`) или напрямую через API.

---

## Перед началом тестирования

| Что открыть | URL |
|---|---|
| Frontend | `http://localhost:5173` |
| Swagger | `http://127.0.0.1:8040/docs` |
| Healthcheck | `http://127.0.0.1:8040/health` |
| Data health | `http://127.0.0.1:8040/health/data` |

---

## TC-01. Backend healthcheck

| Поле | Значение |
|---|---|
| Цель | убедиться, что backend запущен |

## TC-09. Перегруженный сотрудник

| Поле | Значение |
|---|---|
| Цель | проверить расчёт загрузки |
| Тип | API |
| Приоритет | высокий |

**Шаги:** выполнить `GET /employees?department=Quality`, найти сотрудника с высокой `load`, проверить `busy_hours`, `work_hours`, `load`.

**Ожидаемый результат:** `load = busy_hours / work_hours`, сотрудник с высокой загрузкой попадает в `highLoad`.

---

## TC-16. Data-quality

**Шаги:** выполнить `GET /analytics/data-quality`.

**Ожидаемый результат:** endpoint проверяет дубликаты, orphan-записи и некорректные интервалы.

## TC-17. Схемы данных

**Шаги:** выполнить `GET /schemas`.

**Ожидаемый результат:** backend возвращает описание ожидаемых полей для employees, events, hr_profiles, absences.

---

## TC-18. Загрузка CSV/JSON

**Шаги:** подготовить корректный CSV или JSON, выполнить `POST /upload/employees`, затем `GET /health/data`.

**Ожидаемый результат:** файл сохраняется, валидируется и следующий запрос к API использует новый файл.

---

## TC-19. Некорректная загрузка

**Шаги:** загрузить CSV без обязательного поля.

**Ожидаемый результат:** backend отклоняет файл, а невалидный файл не становится активным источником данных.
