# WorkTime Sync

**WorkTime Sync** — веб-система для актуализации рабочего времени сотрудников.  
Проект помогает командам видеть, насколько рабочие графики соответствуют реальной занятости, где возникают встречи вне рабочего времени, кто перегружен, какие данные расходятся с HR-профилем и когда команда реально доступна для совместной работы.

Проект разработан для кейса №3 хакатона **«Весенний код»**:  
**«Система по актуализации рабочего времени сотрудников»**.

---

## Суть проекта

В компаниях рабочее время сотрудников часто хранится в разных местах: HR-системе, календарях, таск-трекерах, табелях и личных договорённостях. Эти данные быстро устаревают: сотрудники переходят на гибридный или удалённый формат, меняют часовой пояс, уходят в отпуск, получают встречи за пределами рабочего времени или оказываются перегружены.

**WorkTime Sync** решает эту проблему как аналитический контур:

1. собирает или имитирует данные о сотрудниках, графиках, встречах, HR-профилях и отсутствиях;
2. сравнивает заявленный график с фактической активностью;
3. рассчитывает метрики актуальности, загрузки, конфликтов и риска;
4. показывает командную доступность;
5. предлагает лучшие окна для встреч;
6. формирует рекомендации и уведомления для руководителей, HR и проектных менеджеров.

Главный вопрос, на который отвечает система:

> Насколько актуальны данные о рабочем времени сотрудников и что нужно обновить, чтобы команда могла планировать работу без конфликтов, перегрузок и ручной координации?

## Что реализовано в MVP

### Пользовательский интерфейс

Frontend реализован на **React + Vite**.

В интерфейсе есть:

- экран входа руководителя отдела;
- дашборд состояния команды;
- список сотрудников;
- карточка сотрудника;
- экран конфликтов;
- командная доступность;
- рекомендации;
- работа с backend через `/api/worktime/overview`;
- fallback на mock-данные, если backend недоступен.

После входа пользователь видит сотрудников только своего отдела.

### Backend и аналитика

Backend реализован на **FastAPI**.

Он умеет:

- читать CSV/JSON-данные из `data/synthetic`;
- валидировать данные через Pydantic;
- нормализовать идентификаторы и даты из CSV;
- рассчитывать актуальность графика;
- считать дни с последнего обновления;
- находить встречи вне рабочего времени;
- считать загрузку сотрудника;
- находить расхождения с HR-профилем;
- рассчитывать интегральный риск;
- строить группы сотрудников;
- объяснять риск по каждому сотруднику;
- формировать рекомендации и уведомления;
- строить командную доступность;
- подбирать лучшие окна для встречи;
- проверять качество данных;
- принимать загрузку новых CSV/JSON-файлов;
- запускать backend-тесты локально и в GitHub Actions.

## Основные данные

Главный источник данных для MVP:

```text

```

Сейчас backend использует:

```text
data/synthetic/employees.csv
data/synthetic/events.csv
data/synthetic/hr_profiles.csv
data/synthetic/absences.csv

```

Текущее наполнение synthetic-данных:

| Dataset | Назначение | Текущее количество |
|---|---|---:|
| `employees.csv` | сотрудники, команды, графики, часовые пояса, формат работы | 25 сотрудников |
| `events.csv` | календарные события и встречи | 60 событий |
| `hr_profiles.csv` | HR-данные для сверки с рабочим графиком | 25 профилей |
| `absences.csv` | отпуска, больничные, личные дни | 3 отсутствия |
| `users.json` | demo-пользователи для входа руководителей | 5 пользователей |

Команды в текущем наборе данных:

```text
Core Platform
Product UI
People Ops
Delivery
Quality
```

`backend/data/` оставлен только как временный fallback. Для командной работы и демо нужно обновлять именно `data/synthetic`.

## Архитектура работы

```text
Пользователь
   ↓
React + Vite frontend
   ↓
POST /auth/login
   ↓
GET /api/worktime/overview?department=<department>
   ↓
FastAPI backend
   ↓
data/synthetic/*.csv
   ↓
analytics.py
   ↓
Метрики, риски, конфликты, доступность, рекомендации
   ↓
Frontend-дашборд и страницы проекта
```

Frontend может работать в двух режимах:

1. **Backend-режим** — данные приходят из FastAPI.
2. **Mock-режим** — данные берутся из `frontend/src/data/mockData.js`, если backend недоступен или включён `VITE_USE_MOCK_DATA=true`.

- часовой пояс;

- актуальность;
- риск;
- количество встреч и конфликтов.

Показывает подробности по конкретному сотруднику:

## Основные метрики

Backend рассчитывает метрики в коде, а не только показывает статичные значения.

### Дни с последнего обновления

```text
days_since_update = DEMO_TODAY - last_update_date
```

### Актуальность графика

```text
schedule_actuality = max(0, 1 - days_since_update / 90)
```

Чем ближе значение к `1`, тем актуальнее график.

### Доля встреч вне рабочего времени

```text
outside_work_ratio = outside_work_events / total_events
```

### Загрузка сотрудника

```text
load = busy_hours / work_hours
```

### Интегральный риск

```text
risk = 0.30 * (1 - schedule_actuality)
     + 0.25 * outside_work_ratio
     + 0.25 * load
     + 0.10 * timezone_mismatch
     + 0.10 * hr_mismatch
```

Статусы риска:

```text
0.00–0.24  -> низкий
0.25–0.49  -> средний
0.50–0.74  -> высокий
0.75–1.00  -> критический
```

Для стабильного демо в backend зафиксирована дата:

```text
DEMO_TODAY = 2026-05-19
```

## Диагностика данных

Для проверки корректности данных есть отдельные endpoint'ы:

```http
GET /health/data
GET /data/source
GET /schemas
GET /analytics/data-quality
```

`/analytics/data-quality` проверяет:

- события без существующего сотрудника;
- отсутствия без существующего сотрудника;
- HR-профили без сотрудника;
- сотрудников без HR-профиля;
- дубликаты id;
- некорректные интервалы событий;
- некорректные интервалы отсутствий.

Это помогает быстро понять, не сломались ли CSV-данные после обновления.

## Как запустить проект локально

### 1. Запустить backend

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
.\.venv\Scripts\python.exe -m uvicorn app.main:app --reload
```

Проверка backend:

```text
http://127.0.0.1:8000/health
http://127.0.0.1:8000/health/data
http://127.0.0.1:8000/docs
```

### 2. Запустить frontend

В отдельном терминале:

```powershell
cd frontend
npm.cmd install
npm.cmd run dev
```

Обычно Vite запускает frontend здесь:

```text
http://localhost:5173
```

### 3. Подключить frontend к backend

Создать файл:

```text
frontend/.env
```

Содержимое:

```env
VITE_API_BASE_URL=http://127.0.0.1:8000
VITE_USE_MOCK_DATA=false
```

После этого frontend будет брать данные из FastAPI.

## GitHub Actions

В проекте используется backend CI:

```text
.github/workflows/backend-ci.yml
```

Он запускает backend-тесты при push и pull request в `main`.

Также в проекте есть workflow деплоя через SSH. Для него должны быть настроены GitHub Secrets:

```text
DEPLOY_KEY
SERVER_HOST
SERVER_USER
```

Если deploy падает из-за SSH-ключа, это не означает ошибку backend-кода. Backend-код проверяется отдельным workflow с тестами.

## Ограничения MVP

Проект является хакатонным MVP, поэтому часть возможностей реализована через synthetic data и demo-логику.

Пока не реализовано как production-система:

- полноценная авторизация с JWT и правами доступа;
- настоящие интеграции с Google Calendar, HRM, Jira/YouTrack;
- база данных PostgreSQL или SQLite вместо CSV/JSON;
- история изменений графиков;
- реальные email/Telegram/Slack-уведомления;
- полноценный AI-chat-ассистент.

Эти пункты логично развивать после защиты.
