# WorkTime Sync

WorkTime Sync — MVP веб-системы для актуализации рабочего времени сотрудников. Проект помогает компании видеть, насколько рабочие графики соответствуют реальной занятости, где есть встречи вне рабочего времени, кто перегружен, где расходятся HR-данные и календарь, а также какие действия нужно выполнить для актуализации данных.

Проект сделан по кейсу №3 хакатона «Весенний код»: **система по актуализации рабочего времени сотрудников**.

## Что делает проект

WorkTime Sync решает проблему устаревших рабочих графиков и ручной сверки данных между календарём, HR-профилем и фактической занятостью.

MVP умеет:

- показывать дашборд состояния команды;
- показывать список сотрудников и их статусы;
- открывать карточку сотрудника;
- сравнивать заявленный график с календарными событиями;
- находить встречи вне рабочего времени;
- выявлять расхождения между HR-профилем и рабочим графиком;
- учитывать временные исключения: отпуск, командировку, личные часы;
- рассчитывать актуальность графика;
- рассчитывать загрузку сотрудника;
- рассчитывать риск неактуальности данных;
- показывать командную доступность;
- подбирать лучшие окна для встречи;
- формировать рекомендации для HR, руководителя и проектного менеджера;
- принимать загрузку JSON/CSV-данных через backend.

Главная идея:

> WorkTime Sync превращает рабочий график из статичной записи в живой аналитический инструмент для планирования, контроля нагрузки и актуализации HR-данных.

## Текущее состояние проекта

В репозитории реализованы две основные части:

```text

backend/  — FastAPI backend с аналитикой, API и загрузкой данных
```

Также есть вспомогательные папки:

```text
docs/          — документация
presentation/  — черновики презентации
tests/         — тест-кейсы
scripts/       — вспомогательные скрипты
```

В корне также есть `deploy.yml` — workflow для автодеплоя через GitHub Actions.

## Frontend

Frontend реализован на React + Vite.

Он показывает:

- дашборд;
- сотрудников;
- профиль сотрудника;
- конфликты;
- командную доступность;
- рекомендации.

Frontend умеет работать в двух режимах:

1. **Mock-режим** — данные берутся из `frontend/src/data/mockData.js`.
2. **Backend-режим** — данные берутся из backend по endpoint:

```http
GET /api/worktime/overview
```

Переключение на backend выполняется через `.env`:

```env
VITE_API_BASE_URL=http://127.0.0.1:8000
VITE_USE_MOCK_DATA=false
```

## Backend

Backend реализован на FastAPI и адаптирован под текущий frontend.

Backend:

- читает данные из JSON/CSV;
- валидирует данные через Pydantic;
- перечитывает данные при каждом запросе;
- считает метрики сотрудников;
- строит summary по команде;
- находит календарные конфликты;
- находит расхождения с HR-данными;
- строит карту командной доступности;
- подбирает лучшие окна встречи;
- формирует рекомендации;
- поддерживает загрузку файлов через API;
- содержит backend-тесты.

## Какие данные используются

Backend использует файлы из папки:

```text
backend/data/
```

Сейчас используются:

```text
employees.json
events.json
hr_profiles.json
absences.json
```

### employees.json

Файл сотрудников и рабочих графиков.

Основные поля:

- `id` — идентификатор сотрудника;
- `name` — имя;
- `team` — отдел / команда;
- `role` — роль;
- `timezone` — часовой пояс;
- `work_start` — начало рабочего дня;
- `work_end` — конец рабочего дня;
- `work_days` — рабочие дни;
- `work_format` — формат работы;
- `last_update_date` — дата последнего обновления графика.

Важно: количество сотрудников не прописано вручную в коде. Backend считает его динамически на основе данных из `employees.json` или `employees.csv`.

### events.json

Файл календарных событий.

Используется для:

- расчёта занятых часов;
- поиска событий вне рабочего времени;
- построения конфликтов;
- расчёта доступности;
- подбора лучших окон встречи.

### hr_profiles.json

Файл HR-профилей.

Используется для сравнения заявленного графика с HR-данными:

- часовой пояс;
- формат работы;
- начало рабочего дня;
- конец рабочего дня.

Если данные расходятся, backend фиксирует `timezone_mismatch` или `hr_mismatch`.

### absences.json

Файл временных исключений:

- командировка;
- личные часы;
- отпуск.

Эти данные учитываются при построении командной доступности и подборе слотов.

## Как подтягиваются сотрудники и отделы

Сотрудники и отделы подтягиваются из данных, а не задаются вручную.

Backend читает `employees.json` или `employees.csv`, затем:

- количество сотрудников считает как длину списка сотрудников;
- отделы/команды берёт из поля `team`;
- группирует сотрудников по `team`;
- считает количество сотрудников, средний риск и конфликты по каждой команде.

Это значит, что если в файл данных добавить новых сотрудников или новые отделы, API начнёт использовать их после следующего запроса.


### GET /api/worktime/overview

Главный endpoint для frontend.

Возвращает объект:

```json
{
  "employees": [],
  "events": [],
  "roadmap": [],
  "summary": {},
  "recommendations": [],
  "bestSlots": [],
  "availability": [],
  "dataMismatches": []
}
```

### GET /employees

Возвращает список сотрудников с рассчитанными метриками.

### GET /employees/frontend

Возвращает сотрудников в формате, удобном для frontend.

### GET /employees/{employee_id}

Возвращает карточку сотрудника:

- данные сотрудника;
- HR-профиль;
- события;
- метрики;
- рекомендации.

### GET /analytics/summary

Возвращает общую аналитику команды:

- количество сотрудников;
- средний риск;
- среднюю актуальность графиков;
- среднюю загрузку;
- календарные конфликты;
- расхождения данных;
- устаревшие графики;
- сотрудников высокого риска;
- распределение по риску;
- команды.

### GET /analytics/conflicts

Возвращает календарные конфликты: встречи и события вне рабочего времени.

### GET /analytics/data-mismatches

Возвращает расхождения между рабочим графиком и HR-профилем.

### GET /analytics/availability

Возвращает карту командной доступности.

### GET /recommendations

Возвращает список рекомендаций.

### GET /meeting-slots

Возвращает лучшие окна для командной встречи.

### POST /upload/{dataset}

Позволяет загрузить JSON/CSV для одного из наборов данных:

```text
employees
events
hr_profiles
absences
```

Если загружен CSV, он используется вместо JSON для этого dataset. Старый файл другого формата удаляется, чтобы не было stale-данных.

## Основные формулы

### Количество дней с последнего обновления

```text
days_since_update = today - last_update_date
```

### Актуальность графика

```text
schedule_actuality = max(0, 1 - days_since_update / 90)
```

### Доля событий вне рабочего времени

```text
outside_work_ratio = outside_work_events / total_events
```

### Загрузка

```text
load = busy_hours / work_hours
```

### Интегральный риск

```text
risk = 0.30 * (1 - schedule_actuality)
     + 0.25 * outside_work_ratio
     + 0.25 * load
     + 0.10 * timezone_mismatch
     + 0.10 * hr_mismatch
```

В backend зафиксированы демо-даты:

```text
DEMO_TODAY = 2026-05-19
DEMO_WEEK_START = 2026-05-18
```

Это сделано, чтобы на защите метрики не менялись каждый день.

В корне репозитория есть `deploy.yml`.

