# WorkTime Sync

WorkTime Sync — MVP веб-системы для актуализации рабочего времени сотрудников. Система помогает понять, насколько рабочие графики соответствуют реальной занятости, где появляются конфликты между календарём и рабочим временем, кто перегружен, какие HR-данные расходятся с фактической активностью и какие действия нужно выполнить для актуализации данных.

Проект делается по кейсу №3 хакатона «Весенний код»: система по актуализации рабочего времени сотрудников.

## Что делает проект

MVP решает задачу диагностики рабочего времени команды:

- показывает список сотрудников и их метрики;
- показывает карточку конкретного сотрудника;
- сравнивает заявленный график с календарными событиями;
- находит встречи вне рабочего времени;
- считает количество дней с последнего обновления графика;
- считает показатель актуальности графика;
- считает долю событий вне рабочего времени;
- считает загрузку сотрудника;
- определяет расхождения между HR-профилем и рабочим графиком;
- рассчитывает интегральный риск неактуальности;
- формирует рекомендации для сотрудника, HR, руководителя и проектного менеджера;
- показывает общую аналитику по команде.

Главная идея проекта: WorkTime Sync превращает рабочий график из статичной записи в живой аналитический инструмент для планирования, контроля нагрузки и актуализации HR-данных.

## Текущее состояние репозитория

На текущем этапе в репозитории есть две основные части:

```text
frontend/ — интерфейс MVP
backend/  — FastAPI backend MVP
```

### Frontend

Frontend реализован на React + Vite.

Он показывает:

- дашборд;
- сотрудников;
- профиль сотрудника;
- конфликты;
- командную доступность;
- рекомендации.

В frontend уже есть API-слой `frontend/src/api/worktimeApi.js`. Он может работать с моковыми данными или обращаться к backend endpoint `/api/worktime/overview`, если отключить mock-режим через `.env`.

### Backend

Backend реализован как минимальный FastAPI MVP.

Он умеет:

- читать тестовые данные из JSON;
- читать JSON или CSV через универсальную функцию `load_table()`;
- отдавать список сотрудников с метриками;
- отдавать карточку сотрудника;
- отдавать общую аналитику по команде;
- считать ключевые метрики: актуальность, конфликты, загрузку и риск.

Основные backend endpoint'ы:

```http
GET /
GET /employees
GET /employees/{employee_id}
GET /analytics/summary
```

## Какие данные используются

В проекте используются синтетические данные.

### Backend-данные

Backend читает данные из папки:

```text

```

Файлы:

```text
backend/data/employees.json

backend/data/hr_profiles.json
```

#### employees.json

Содержит сотрудников и их заявленные рабочие графики.

Основные поля:

- `id` — идентификатор сотрудника;
- `name` — имя сотрудника;
- `team` — команда;
- `role` — должность;
- `timezone` — заявленный часовой пояс;
- `work_start` — начало рабочего дня;
- `work_end` — конец рабочего дня;
- `work_days` — рабочие дни;
- `work_format` — формат работы: office, remote, hybrid;
- `last_update_date` — дата последнего обновления графика.

#### events.json

Содержит календарные события сотрудников:

- `id` — идентификатор события;
- `employee_id` — сотрудник, к которому относится событие;
- `title` — название события;
- `start_datetime` — начало события;
- `end_datetime` — конец события;
- `source` — источник данных;
- `type` — тип события: meeting или focus.

Эти события используются для поиска встреч вне рабочего времени и расчёта занятости.

#### hr_profiles.json

Содержит HR-профиль сотрудника:

- `employee_id`;
- `hr_timezone`;
- `hr_work_format`;
- `hr_work_start`;
- `hr_work_end`.

Эти данные сравниваются с заявленным рабочим графиком сотрудника. Если есть расхождения, система повышает риск и формирует рекомендацию по сверке данных.

## Какие API реализованы

### GET /

Проверочный endpoint backend.

Возвращает название сервиса, ссылку на документацию и список основных endpoint'ов.

### GET /employees

Возвращает список сотрудников с рассчитанными метриками.

Используется для общего анализа команды и отображения сотрудников.

### GET /employees/{employee_id}

Возвращает карточку сотрудника:

- данные сотрудника;
- HR-профиль;
- календарные события;
- рассчитанные метрики;
- рекомендации.

### GET /analytics/summary

Возвращает общую аналитику по команде:

- количество сотрудников;
- средний риск;
- среднюю актуальность графиков;
- среднюю загрузку;
- общее количество конфликтов;
- распределение по уровням риска;
- аналитику по командам.

## Основные формулы

### Количество дней с последнего обновления

```text
days_since_update = today - last_update_date
```

### Показатель актуальности графика

```text
schedule_actuality = max(0, 1 - days_since_update / 90)
```

Если показатель близок к 1, график считается актуальным. Если близок к 0, сотруднику нужно подтвердить или обновить рабочее время.

### Доля событий вне рабочего времени

```text
outside_work_ratio = outside_work_events / total_events
```

Если доля высокая, значит встречи часто назначаются вне рабочего графика.

### Уровень загрузки

```text
load = busy_hours / work_hours
```

Если показатель выше 0.8, сотрудник считается близким к перегрузке.

### Интегральный риск неактуальности

```text
risk = 0.30 * (1 - schedule_actuality)
     + 0.25 * outside_work_ratio
     + 0.25 * load
     + 0.10 * timezone_mismatch
     + 0.10 * hr_mismatch
```

Где:

- `schedule_actuality` — актуальность графика;
- `outside_work_ratio` — доля событий вне рабочего времени;
- `load` — загрузка сотрудника;
- `timezone_mismatch` — расхождение часового пояса;
- `hr_mismatch` — расхождение с HR-данными.

## Как запустить backend

```bash
cd backend
python -m venv .venv
```

Активация окружения:

```bash
# Linux / macOS
source .venv/bin/activate

# Windows
.venv\Scripts\activate
```

Установка зависимостей:

```bash
pip install -r requirements.txt
```

Запуск:

```bash
uvicorn app.main:app --reload
```

После запуска открыть:

```text
http://127.0.0.1:8000/docs
```

## Как запустить frontend

```bash
cd frontend
npm install
npm run dev
```

Обычно Vite запускается по адресу:

```text
http://localhost:5173
```

## Как подключить frontend к backend

В `frontend/` создать файл `.env`:

```bash
VITE_USE_MOCK_DATA=false
VITE_API_BASE_URL=http://127.0.0.1:8000
```

Важно: текущий frontend API-слой ожидает endpoint:

```http
GET /api/worktime/overview
```

А текущий backend реализует:

```http
GET /employees
GET /employees/{id}
GET /analytics/summary
```

Поэтому для полной связки frontend + backend нужно либо:

1. добавить в backend endpoint `/api/worktime/overview`, который вернёт агрегированный объект для frontend;
2. либо изменить `frontend/src/api/worktimeApi.js`, чтобы он обращался к текущим backend endpoint'ам.

## Что показывает MVP

MVP показывает ценность системы на синтетических данных:

- у кого график давно не обновлялся;
- у кого есть встречи вне рабочего времени;
- у кого высокая загрузка;
- где HR-данные не совпадают с заявленным графиком;
- какой риск неактуальности у сотрудника;
- какие рекомендации нужно выполнить;
- какое общее состояние команды.

## Что пока отсутствует или требует уточнения

- Нет полной интеграции frontend и backend через единый endpoint `/api/worktime/overview`.
- Нет базы данных.
- Нет реальных интеграций с календарём, HRM, таск-трекером и табелем.
- Нет авторизации и разделения прав пользователей.
- Нет отдельного AI-чат-ассистента.
- Нет отдельного endpoint для командной доступности в backend.
- Нет endpoint для подбора лучших окон встречи в backend.
- Нет endpoint для списка всех рекомендаций в backend.
- Нет загрузки файлов через UI.
- Нет автоматических тестов как исполняемого набора, есть только подготовленные сценарии тестирования.
