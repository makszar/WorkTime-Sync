# WorkTime Sync

**WorkTime Sync** — веб-система для актуализации рабочего времени сотрудников. Проект помогает видеть, насколько рабочие графики соответствуют реальной занятости, где возникают встречи вне рабочего времени, кто перегружен, какие данные расходятся с HR-профилем и когда команда реально доступна для совместной работы.

Проект разработан для кейса №3 хакатона **«Весенний код»**: **«Система по актуализации рабочего времени сотрудников»**.

---

## Суть проекта

В компаниях рабочее время сотрудников часто хранится в разных местах: HR-системе, календарях, таск-трекерах, табелях и личных договорённостях. Эти данные быстро устаревают: сотрудники переходят на гибридный или удалённый формат, меняют часовой пояс, уходят в отпуск, получают встречи за пределами рабочего времени или оказываются перегружены.

**WorkTime Sync** решает эту проблему как аналитический контур:

1. собирает или имитирует данные о сотрудниках, графиках, встречах, HR-профилях и отсутствиях;
2. сравнивает заявленный график с фактической активностью;
3. рассчитывает метрики актуальности, загрузки, конфликтов и риска;
4. показывает командную доступность;
5. предлагает лучшие окна для встреч;
6. формирует рекомендации и уведомления для руководителей, HR и проектных менеджеров.

Главный вопрос, на который отвечает система:

> Насколько актуальны данные о рабочем времени сотрудников и что нужно обновить, чтобы команда могла планировать работу без конфликтов, перегрузок и ручной координации?

## Что реализовано в MVP сейчас

### Frontend

Frontend реализован на **React + Vite**.

Уже есть:

- экран входа руководителя отдела;
- дашборд состояния отдела;
- список сотрудников отдела;
- карточка сотрудника;
- экран конфликтов;
- командная доступность;
- рекомендации;
- работа с backend через `/api/worktime/overview`;
- fallback на mock-данные, если backend недоступен.

После входа руководитель отдела видит сотрудников только своего отдела.

### Backend

Backend реализован на **FastAPI**.

Уже есть:

- чтение CSV/JSON-данных из `data/synthetic`;
- Pydantic-валидация данных;
- нормализация идентификаторов и дат из CSV;
- расчёт актуальности графика;
- расчёт дней с последнего обновления;
- поиск встреч вне рабочего времени;
- расчёт загрузки сотрудника;
- поиск расхождений с HR-профилем;
- расчёт интегрального риска;
- группы сотрудников;
- объяснение риска по сотруднику;
- рекомендации и уведомления;
- командная доступность и лучшие окна встречи;
- диагностика качества данных;
- загрузка новых CSV/JSON-файлов;
- backend-тесты и GitHub Actions CI.

## Новое направление после первого митапа

После первого митапа команда решила развивать проект не только как аналитический дашборд, а как **систему взаимодействия ролей**.

Новая логика:

```text
система находит проблему
   ↓
руководитель или HR видит риск
   ↓
создаётся задача / запрос сотруднику
   ↓
сотрудник видит задачу в личном кабинете
   ↓
сотрудник подтверждает график или оставляет комментарий
   ↓
руководитель / HR видит обновлённый статус
```

То есть WorkTime Sync должен не просто показывать проблему, а помогать закрывать её через подтверждение графика и задачи.

---

## Планируемая ролевая модель

| Роль | Область доступа | Что видит | Что может делать |
|---|---|---|---|
| Полный руководитель | вся компания | все отделы, всех сотрудников, общую аналитику, риски и сравнение отделов | смотреть глобальную картину, фильтровать отделы, контролировать проблемные зоны |
| Руководитель отдела | свой отдел | сотрудников своего отдела, конфликты, загрузку, доступность, рекомендации | создавать задачи сотрудникам, запрашивать подтверждение графика, контролировать выполнение |
| HR | все отделы с HR-фокусом | устаревшие графики, HR-расхождения, timezone-конфликты, сотрудников без подтверждения | создавать HR-задачи, запрашивать подтверждение данных, контролировать качество HR-профилей |
| Сотрудник | только свой профиль | свой график, свои события, свой риск, свои задачи и рекомендации | подтвердить график, принять/отклонить задачу, оставить комментарий, запросить изменение |

Важно: роли должны влиять на **доступ и действия**, но не должны ломать текущую аналитику и текущие endpoint’ы.

## Архитектура работы

Текущая архитектура:

```text
Пользователь
   ↓
React + Vite frontend
   ↓
POST /auth/login
   ↓
GET /api/worktime/overview?department=<department>
   ↓
FastAPI backend
   ↓
data/synthetic/*.csv
   ↓
analytics.py + data_quality.py
   ↓
Метрики, риски, конфликты, доступность, рекомендации
```

Планируемое расширение:

```text
Пользователь входит в систему
   ↓
Backend возвращает role + scope + department + employee_id
   ↓
Frontend выбирает интерфейс под роль
   ↓
Backend отдаёт данные согласно scope
   ↓
Руководитель/HR создаёт задачу
   ↓
Сотрудник подтверждает или отклоняет задачу
   ↓
Статус возвращается руководителю/HR
```

## Основные метрики

| Метрика | Формула / логика | Зачем нужна |
|---|---|---|
| `days_since_update` | `DEMO_TODAY - last_update_date` | показывает, как давно сотрудник подтверждал график |
| `schedule_actuality` | `max(0, 1 - days_since_update / 90)` | оценивает актуальность графика |
| `outside_work_ratio` | `outside_work_events / total_events` | показывает долю встреч вне рабочего времени |
| `load` | `busy_hours / work_hours` | показывает загрузку сотрудника |
| `timezone_mismatch` | сравнение timezone сотрудника и HR-профиля | выявляет конфликт часового пояса |
| `hr_mismatch` | сравнение формата и рабочих часов с HR-профилем | выявляет расхождение HR-данных |

Актуальность графика остаётся одинаковой для всех отделов:

```text
A = max(0, 1 - days_since_update / 90)
```

Итоговый риск планируется считать по одной структуре формулы, но с весами под отдел:

```text
R = wA * (1 - A)
  + wC * C
  + wL * L
  + wT * timezone_mismatch
  + wH * hr_mismatch
```

| Отдел | `wA` | `wC` | `wL` | `wT` | `wH` |
|---|---:|---:|---:|---:|---:|
| Core Platform | 0.25 | 0.25 | 0.30 | 0.10 | 0.10 |
| Product UI | 0.25 | 0.30 | 0.25 | 0.10 | 0.10 |
| People Ops | 0.30 | 0.15 | 0.20 | 0.10 | 0.25 |
| Delivery | 0.20 | 0.30 | 0.30 | 0.10 | 0.10 |
| Quality | 0.20 | 0.25 | 0.35 | 0.10 | 0.10 |

## Как запустить проект локально

### Backend

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
.\.venv\Scripts\python.exe -m uvicorn app.main:app --reload
```

Проверка:

```text
http://127.0.0.1:8000/health
http://127.0.0.1:8000/health/data
http://127.0.0.1:8000/docs
```

### Frontend

```powershell
cd frontend
npm.cmd install
npm.cmd run dev
```

Подключение frontend к backend:

```env
VITE_API_BASE_URL=http://127.0.0.1:8000
VITE_USE_MOCK_DATA=false
```

- полноценная авторизация с JWT и правами доступа;
- личные кабинеты сотрудников;

- подтверждение графика через UI;
- настоящие интеграции с Google Calendar, HRM, Jira/YouTrack;
- база данных PostgreSQL или SQLite вместо CSV/JSON;
- история изменений графиков;
- реальные email/Telegram/Slack-уведомления;
- полноценный AI-chat-ассистент.
