# WorkTime Sync

**WorkTime Sync** — веб-система для актуализации рабочего времени сотрудников. Проект помогает видеть реальные рабочие графики, конфликты с календарём, перегрузки, расхождения с HR-данными, статусы подтверждения графиков и задачи, которые нужно выполнить для устранения проблем.

Проект разработан для кейса №3 хакатона **«Весенний код»**: **«Система по актуализации рабочего времени сотрудников»**.

---

## 1. Суть проекта

В компаниях рабочее время сотрудников часто хранится в разных местах: HR-системе, календарях, таск-трекерах, табелях и личных договорённостях. Эти данные быстро устаревают: сотрудники меняют формат работы, переходят на удалёнку или гибрид, работают из другого часового пояса, уходят в отпуск или получают встречи вне рабочего времени.

**WorkTime Sync** решает эту проблему как аналитический и workflow-контур:

1. собирает или имитирует данные о сотрудниках, графиках, встречах, HR-профилях и отсутствиях;
2. сравнивает заявленный график с фактической активностью;
3. рассчитывает актуальность графика, загрузку, конфликты и риск;
4. показывает проблемные зоны руководителям и HR;
5. позволяет создавать задачи сотрудникам;
6. позволяет сотрудникам подтверждать график или отвечать на задачи;
7. помогает снижать конфликты, переработки и ручную координацию.

Главная логика:

```text
аналитика → задача → подтверждение сотрудником → актуализация графика
```

## 3. Ролевая модель

| Роль | Scope | Что видит | Что может делать |
|---|---|---|---|
| `executive` | `all` | всю компанию, все отделы, всех сотрудников | смотреть общую аналитику, контролировать отделы, создавать задачи любому сотруднику |
| `hr` | `all` | всех сотрудников, HR-расхождения, подтверждения | создавать HR-задачи, проверять графики и HR-данные |
| `department_manager` | `department` | только сотрудников своего отдела | создавать задачи сотрудникам своего отдела, контролировать графики, встречи и нагрузку |
| `employee` | `self` | только свой профиль и свои задачи | подтверждать график, подтверждать/отклонять задачи, оставлять комментарии |

## 6. Задачи в системе

Сейчас реализован общий механизм задач.

| Тип | Что значит |
|---|---|
| `confirm_schedule` | подтвердить рабочий график |
| `review_hr_profile` | проверить HR-профиль |
| `review_load` | проверить нагрузку |

Планируемые типы задач по встречам:

| Тип | Что значит |
|---|---|
| `meeting_confirmation` | подтвердить участие во встрече |
| `reschedule_meeting` | согласовать перенос встречи |
| `meeting_outside_work_approval` | согласовать встречу вне рабочего времени |

Важно: сама встреча должна храниться в `events.csv`, а задача — это действие вокруг этой встречи.

```text
events.csv  → сама встреча
tasks.json  → подтвердить / перенести / согласовать эту встречу
```

## 7. Формулы и метрики

Актуальность графика остаётся единой для всех отделов:

```text
A = max(0, 1 - days_since_update / 90)
```

Риск рассчитывается по общей структуре, но с весами отдела:

```text
R = wA * (1 - A)
  + wC * C
  + wL * L
  + wT * timezone_mismatch
  + wH * hr_mismatch
```

| Отдел | `wA` | `wC` | `wL` | `wT` | `wH` |
|---|---:|---:|---:|---:|---:|
| Core Platform | 0.25 | 0.25 | 0.30 | 0.10 | 0.10 |
| Product UI | 0.25 | 0.30 | 0.25 | 0.10 | 0.10 |
| People Ops | 0.30 | 0.15 | 0.20 | 0.10 | 0.25 |
| Delivery | 0.20 | 0.30 | 0.30 | 0.10 | 0.10 |
| Quality | 0.20 | 0.25 | 0.35 | 0.10 | 0.10 |

## 8. Backend API

| Endpoint | Назначение |
|---|---|
| `POST /auth/login` | demo-вход |
| `GET /health` | проверка backend |
| `GET /health/data` | проверка активных данных |
| `GET /api/worktime/overview` | основной endpoint frontend |
| `GET /employees/me` | личный кабинет сотрудника на уровне API |
| `GET /employees/{employee_id}/risk-explanation` | объяснение риска |
| `PATCH /employees/{employee_id}/confirm-schedule` | подтверждение графика |
| `GET /analytics/company` | аналитика всей компании |
| `GET /analytics/hr-dashboard` | HR-dashboard API |
| `GET /tasks` | список задач |
| `GET /tasks/my` | задачи сотрудника |
| `POST /tasks` | создать задачу |
| `GET /tasks/{task_id}` | открыть задачу |
| `PATCH /tasks/{task_id}/status` | изменить статус задачи |

Примеры:

```text
GET /api/worktime/overview?user_id=u0
GET /api/worktime/overview?user_id=u1
GET /api/worktime/overview?user_id=emp5
GET /tasks?user_id=u1
GET /tasks/my?user_id=emp5
```

## 12. Ограничения MVP

- нет production JWT-авторизации;
- пароли demo-пользователей лежат в JSON;
- нет базы данных;
- задачи и подтверждения хранятся в JSON;
- нет реальных интеграций с календарём, HRM и таск-трекером;
- нет реальных push/email/Telegram-уведомлений;
- frontend ещё не визуализирует все backend-возможности V2.2;
- история изменений графиков пока минимальная.
