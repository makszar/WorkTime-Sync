# WorkTime Sync

**WorkTime Sync** — веб-система для актуализации рабочего времени сотрудников. Проект помогает руководителям, HR и PM видеть, насколько рабочие графики соответствуют реальной занятости, где возникают встречи вне рабочего времени, кто перегружен, какие данные расходятся с HR-профилем и когда команда реально доступна для совместной работы.

Проект разработан для кейса №3 хакатона **«Весенний код»**: **«Система по актуализации рабочего времени сотрудников»**.

---

## Суть проекта

Рабочее время сотрудников часто хранится в разных местах: HR-системе, календарях, таск-трекерах, табелях и личных договорённостях. Эти данные быстро устаревают: сотрудники переходят на гибридный или удалённый формат, меняют часовой пояс, уходят в отпуск, получают встречи за пределами рабочего времени или оказываются перегружены.

**WorkTime Sync** решает эту проблему как аналитический контур:

1. собирает или имитирует данные о сотрудниках, графиках, встречах, HR-профилях и отсутствиях;
2. сравнивает заявленный график с фактической активностью;
3. рассчитывает метрики актуальности, загрузки, конфликтов и риска;
4. показывает командную доступность;
5. предлагает лучшие окна для встреч;
6. формирует рекомендации и уведомления для руководителей, HR и проектных менеджеров.

Главный вопрос, на который отвечает система:

> Насколько актуальны данные о рабочем времени сотрудников и что нужно обновить, чтобы команда могла планировать работу без конфликтов, перегрузок и ручной координации?

## Что реализовано в MVP

### Пользовательский интерфейс

Frontend реализован на **React + Vite**.

В интерфейсе есть:

- экран входа руководителя отдела;
- дашборд состояния команды;
- список сотрудников;
- карточка сотрудника;
- экран конфликтов;
- командная доступность;
- рекомендации;
- работа с backend через `/api/worktime/overview`;
- fallback на mock-данные, если backend недоступен.

После входа пользователь видит сотрудников только своего отдела.

### Backend и аналитика

Backend реализован на **FastAPI**.

Он умеет:

- читать CSV/JSON-данные из `data/synthetic`;
- валидировать данные через Pydantic;
- нормализовать идентификаторы и даты из CSV;
- рассчитывать актуальность графика;
- считать дни с последнего обновления;
- находить встречи вне рабочего времени;
- считать загрузку сотрудника;
- находить расхождения с HR-профилем;
- рассчитывать интегральный риск;
- строить группы сотрудников;
- объяснять риск по каждому сотруднику;
- формировать рекомендации и уведомления;
- строить командную доступность;
- подбирать лучшие окна для встречи;
- проверять качество данных;
- принимать загрузку новых CSV/JSON-файлов;
- запускать backend-тесты локально и в GitHub Actions.

## Основные данные

Главный источник данных для MVP:

```text
data/synthetic/
```

| Dataset | Назначение | Текущее количество |
|---|---|---:|
| `employees.csv` | сотрудники, команды, графики, часовые пояса, формат работы | 25 сотрудников |
| `events.csv` | календарные события и встречи | 60 событий |
| `hr_profiles.csv` | HR-данные для сверки с рабочим графиком | 25 профилей |
| `absences.csv` | отпуска, больничные, личные дни | 3 отсутствия |
| `users.json` | demo-пользователи для входа руководителей | 5 пользователей |

Команды в текущем наборе данных:

| Команда / отдел | Количество сотрудников | Что демонстрирует |
|---|---:|---|
| Core Platform | 5 | распределённая backend/platform-команда с рисками по часовым поясам и устаревшим графикам |
| Product UI | 5 | frontend/design-команда с UI-встречами и пересечениями по графику |
| People Ops | 5 | HR-команда, которая важна для сверки HR-профилей |
| Delivery | 5 | команда проектного управления и релизной координации |
| Quality | 5 | QA-команда с нагрузкой, тестированием и возможными конфликтами по времени |

`backend/data/` оставлен только как временный fallback. Для командной работы и демо нужно обновлять именно `data/synthetic`.

## Основные метрики

| Метрика | Формула / логика | Зачем нужна |
|---|---|---|
| `days_since_update` | `DEMO_TODAY - last_update_date` | показывает, как давно сотрудник подтверждал график |
| `schedule_actuality` | `max(0, 1 - days_since_update / 90)` | оценивает актуальность графика |
| `outside_work_ratio` | `outside_work_events / total_events` | показывает долю встреч вне рабочего времени |
| `load` | `busy_hours / work_hours` | показывает загрузку сотрудника |
| `timezone_mismatch` | сравнение timezone сотрудника и HR-профиля | выявляет конфликт часового пояса |
| `hr_mismatch` | сравнение формата и рабочих часов с HR-профилем | выявляет расхождение HR-данных |
| `risk` | интегральная формула риска | показывает, кого проверять в первую очередь |

```text
risk = 0.30 * (1 - schedule_actuality)
     + 0.25 * outside_work_ratio
     + 0.25 * load
     + 0.10 * timezone_mismatch
     + 0.10 * hr_mismatch
```

Для стабильного демо в backend зафиксирована дата:

```text
DEMO_TODAY = 2026-05-19
```

| Endpoint | Назначение |
|---|---|
| `POST /auth/login` | demo-вход руководителя отдела |

## Диагностика данных

```http
GET /health/data
GET /data/source
GET /schemas
GET /analytics/data-quality
```

`/analytics/data-quality` проверяет orphan events/absences/HR profiles, сотрудников без HR-профиля, дубликаты id, некорректные интервалы событий и отсутствий.

## Как запустить проект локально

### Backend

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
.\.venv\Scripts\python.exe -m uvicorn app.main:app --reload
```

### Frontend

```powershell
cd frontend
npm.cmd install
npm.cmd run dev
```

### Подключить frontend к backend

Создать `frontend/.env`:

```env
VITE_API_BASE_URL=http://127.0.0.1:8000
VITE_USE_MOCK_DATA=false
```

## Ограничения MVP

Пока не реализовано как production-система: полноценная авторизация с JWT, реальные интеграции с Google Calendar/HRM/Jira, база данных PostgreSQL/SQLite, история изменений графиков, реальные email/Telegram/Slack-уведомления, полноценный AI-chat-ассистент.

## Возможное развитие

1. заменить CSV/JSON на SQLite или PostgreSQL;
2. добавить историю изменений графиков;
3. подключить реальные календари и HRM;
4. добавить роли и права доступа;
5. отправлять уведомления в email/Telegram/Slack;
6. сделать AI-ассистента поверх рассчитанных метрик;
7. добавить экспорт отчётов;
8. расширить frontend-экраны для `groups`, `notifications` и `data-quality`.
