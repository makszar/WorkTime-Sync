# Черновик презентации WorkTime Sync

## Слайд 1. Титульный слайд

**WorkTime Sync**  
Актуализация рабочего времени сотрудников

Кейс №3 хакатона «Весенний код»

> Рабочий график должен стать живым источником планирования, а не устаревшей записью в HR-системе.

---

## Слайд 2. Проблема

| Проблема | Последствие |
|---|---|
| устаревшие графики | руководитель не понимает, когда сотрудник реально доступен |
| разные часовые пояса | встречи попадают в неудобное или нерабочее время |
| встречи вне рабочего времени | растёт нагрузка и недовольство сотрудников |
| разные данные в HR и календаре | HR вынужден сверять всё вручную |
| отсутствия не учитываются | встречи ставятся во время отпуска, больничного или day off |
| нет карты доступности | PM тратит время на ручной подбор окна |

Главный вопрос кейса: что обновить, чтобы планировать работу без конфликтов и лишней ручной координации?

---

## Слайд 3. Идея решения

**WorkTime Sync** собирает данные о рабочем времени, сравнивает их между собой и показывает, что нужно обновить.

Система анализирует профиль сотрудника, рабочие часы, рабочие дни, часовой пояс, формат работы, календарные события, HR-профиль, отсутствия, загрузку и дату последнего обновления графика.

---

## Слайд 4. Пользователи и роли

| Роль | Что делает в системе |
|---|---|
| Руководитель отдела | видит сотрудников своего отдела, риски, конфликты и доступность |
| HR | проверяет HR-профили, форматы работы и часовые пояса |
| PM | выбирает лучшие окна для встреч и переносит конфликтные события |
| Сотрудник | подтверждает или обновляет свой рабочий график |
| Администратор | загружает данные и управляет источниками |
| Аналитик | анализирует группы сотрудников, риски и тренды |

## Слайд 10. Метрики

| Метрика | Формула / логика |
|---|---|
| Дни с обновления | `DEMO_TODAY - last_update_date` |
| Актуальность | `max(0, 1 - days_since_update / 90)` |
| Доля конфликтов | `outside_work_events / total_events` |
| Загрузка | `busy_hours / work_hours` |
| Timezone mismatch | сравнение timezone сотрудника и HR |
| HR mismatch | сравнение формата и рабочих часов |
| Риск | взвешенная сумма факторов |

```text
risk = 0.30 * (1 - schedule_actuality)
     + 0.25 * outside_work_ratio
     + 0.25 * load
     + 0.10 * timezone_mismatch
     + 0.10 * hr_mismatch
```

## Слайд 11. Объяснение риска

Endpoint `GET /employees/{employee_id}/risk-explanation` показывает итоговый риск, статус, формулу, факторы, вклад факторов и рекомендуемые действия.

---

## Слайд 12. Конфликты

| Ситуация | Как система реагирует |
|---|---|
| встреча позже `work_end` | помечает как конфликт |
| встреча раньше `work_start` | помечает как конфликт |
| событие в нерабочий день | помечает как конфликт |
| регулярные встречи вне графика | повышает риск |
| конфликт у перегруженного сотрудника | формирует рекомендацию перенести встречу |

---

## Слайд 13. Командная доступность

Система учитывает рабочие дни, рабочие часы, календарные события, focus-блоки, отсутствия, перегруз и часовые пояса. На основе этой карты система предлагает лучшие окна встречи.

---

## Слайд 14. Рекомендации и уведомления

| Тип | Пример |
|---|---|
| График | попросить сотрудника подтвердить рабочий график |
| Часовой пояс | проверить timezone сотрудника |
| Нагрузка | не назначать новые встречи перегруженному сотруднику |
| Встреча | перенести событие вне рабочего времени |
| HR | сверить данные HR-профиля |
| PM | выбрать рекомендуемый слот встречи |

---

## Слайд 15. Диагностика качества данных

Endpoint `GET /analytics/data-quality` проверяет события без сотрудника, отсутствия без сотрудника, HR-профили без сотрудника, сотрудников без HR-профиля, дубликаты id и некорректные интервалы.

## Слайд 20. Ограничения и развитие

Ограничения MVP: нет production-авторизации с JWT, реальных интеграций, полноценной БД, реальной отправки уведомлений и AI-chat ассистента.

Развитие: PostgreSQL/SQLite, реальные источники данных, права доступа, история изменений графиков, email/Telegram/Slack-уведомления, AI-ассистент.
