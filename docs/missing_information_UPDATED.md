# WorkTime Sync — финальный список недочётов и задач после проверки V2.9

Дата проверки: 2026-05-25  
Репозиторий: `makszar/WorkTime-Sync`  
Текущее состояние: backend `2.9.0`, frontend `v2.9`, MVP в целом реализован, но перед демонстрацией нужно закрыть UX-, data- и deploy-недочёты.

---

## 1. Краткий итог

Проект уже соответствует основной логике кейса:

```text
данные → аналитика → задача → ответ сотрудника → применение решения → обновление данных
```

Реализованы роли, demo-login, demo-token, role-based доступ, личный кабинет, HR/Executive dashboard, задачи, подтверждение графика, задачи по встречам, `related_event_id`, применение решения через `/api/tasks/{task_id}/apply`, автогенерация задач по конфликтным встречам и `/api`-алиасы для deploy.

Главная задача сейчас — не добавлять большую новую функциональность, а довести MVP до понятной и стабильной демонстрации.

# 2. Критичные недочёты перед показом

## 2.1. Кнопки задач недостаточно понятные

### Что сейчас
В `TaskCard.jsx` кнопки универсальные:

```text
[Подтвердить]
[Отклонить]
[Запросить перенос]
[Применить решение]
```

### Проблема
Для задач по встречам пользователь не понимает, что именно подтверждает. Для `reschedule_meeting` нужно не “Подтвердить”, а “Согласен на перенос”. Для `meeting_confirmation` — “Подтверждаю участие”. Для `meeting_outside_work_approval` — “Согласен на встречу вне рабочего времени”.

### Кто делает
Frontend.

### Что сделать
В `frontend/src/components/TaskCard.jsx` добавить выбор текста кнопки по `task.type`.

Рекомендуемые подписи:

| Тип задачи | Основная кнопка | Отказ | Альтернатива |
|---|---|---|---|
| `confirm_schedule` | Подтверждаю график | Нужно изменить график | — |
| `meeting_confirmation` | Подтверждаю участие | Не могу участвовать | — |
| `reschedule_meeting` | Согласен на перенос | Не согласен | Предложить другое время |
| `meeting_outside_work_approval` | Согласен на встречу вне рабочего времени | Не согласен | Предложить перенос |

### Критерий готовности
Сотрудник без объяснений понимает, что делает каждая кнопка.

## 2.2. Нет явной кнопки “Предложить другое время” для встречи вне рабочего времени

### Что сейчас
Для `reschedule_meeting` есть запрос переноса. Для `meeting_outside_work_approval` UX не показывает полноценный сценарий “Предложить другое время”.

### Почему важно
По кейсу система должна помогать переносить встречи вне рабочего времени. Если встреча стоит после графика, сотрудник должен не только согласиться или отказаться, но и предложить рабочее окно.

### Кто делает
Frontend + Backend проверить.

### Что сделать во frontend
Показывать поля нового времени не только для `reschedule_meeting`, но и для `meeting_outside_work_approval`.

Payload при предложении времени:

```json
{
  "status": "reschedule_requested",
  "employee_comment": "Предлагаю перенести на рабочее время.",
  "suggested_start_datetime": "2026-05-22T12:00:00",
  "suggested_end_datetime": "2026-05-22T13:00:00"
}
```

### Что проверить в backend
Backend уже умеет применять перенос для `meeting_outside_work_approval`, если есть `suggested_start_datetime` и `suggested_end_datetime`. Нужно проверить это тестом.

## 2.3. Поля даты/времени неудобные

### Что сейчас
Пользователь вводит время вручную в ISO-формате:

```text
2026-05-22T12:00:00
```

### Проблема
Это неудобно для обычного пользователя и легко даёт ошибки.

### Кто делает
Frontend.

### Что сделать
Заменить текстовые поля на:

```html
<input type="datetime-local" />
```

Потом преобразовывать значение в ISO-строку, совместимую с backend.

## 2.4. В календаре/событиях может показываться только день недели, а не дата

### Что сейчас
В интерфейсе местами используется:

```js
event.day || event.start_datetime
```

или в форме выбора встречи:

```js
event.title - event.day event.time
```

### Проблема
Если пользователь видит только “Пн 10:00–11:00”, непонятно, о какой дате идёт речь. Для переноса встреч это критично.

### Кто делает
Frontend + Data проверить.

### Что сделать во frontend
Во всех карточках встречи показывать:

```text
22.05.2026, Пт, 12:00–13:00
```

Использовать `start_datetime` и `end_datetime`.

### Что проверить в данных
У всех событий должны быть `start_datetime` и `end_datetime`. Если дат нет — добавить их в `events.csv/json`.

## 2.5. Сотрудник должен видеть не только задачи “что подтвердить”, но и активные согласованные встречи/решения

### Что сейчас
Employee cabinet показывает задачи и события, но UX может восприниматься как список того, что нужно подтвердить/перенести.

### Проблема
Сотрудник должен понимать:

```text
что требует действия
что уже подтверждено
что ожидает руководителя
что закрыто
какие встречи запланированы
```

### Кто делает
Frontend, backend менять только при нехватке данных.

### Что сделать
В `EmployeeCabinet.jsx` разделить блоки:

1. Требуют моего ответа.
2. Запланированные встречи.
3. Подтверждённые/согласованные задачи.
4. Запрошенные переносы.
5. Закрытые решения.

Статусы показывать человеку понятными фразами:

| Статус | Текст для сотрудника |
|---|---|
| `pending` | Нужно ответить |
| `confirmed` | Вы подтвердили, ожидает применения руководителем |
| `reschedule_requested` | Вы предложили другое время |
| `rejected` | Вы отклонили, ожидает решения руководителя |
| `done` | Решение применено / задача закрыта |


В MVP уведомления реализованы через задачи. В интерфейсе лучше писать “Задачи и уведомления”, а не просто “уведомления”.

# 4. Проблемы запуска и реализации

## 4.1. Ошибка 405 при кнопках

Возможные причины:

1. frontend стучится не в тот URL;
2. backend старой версии;
3. nginx не проксирует `/api`;
4. запрос идёт методом GET вместо POST/PATCH;
5. браузер использует старый bundle;
6. сервер не перезапущен после обновления.

Проверить в DevTools → Network:

```text
POST /api/tasks
PATCH /api/tasks/{id}/status
PATCH /api/employees/{id}/confirm-schedule
POST /api/tasks/{id}/apply
```

## 4.2. Домен открывается, но показывает Not Found

Причина: сервер не отдаёт React `dist`.

Nginx должен иметь:

```nginx
location / {
  try_files $uri $uri/ /index.html;
}

location /api/ {
  proxy_pass http://127.0.0.1:8000/api/;
}
```

## 4.3. `VITE_API_BASE_URL`

Для deploy на одном домене можно оставить пустым:

```env
VITE_API_BASE_URL=
```

Для локального запуска:

```env
VITE_API_BASE_URL=http://127.0.0.1:8000
```

## 4.4. Права на запись данных

Для `save_tasks`, `save_events`, `save_schedule_confirmations` backend должен иметь права записи в папку данных.

## 4.5. Формат дат

Backend ждёт ISO datetime. Если frontend перейдёт на `datetime-local`, нужно проверить преобразование.
