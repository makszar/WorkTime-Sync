# Missing information WorkTime Sync — финальная инструкция для закрытия MVP

Документ фиксирует **финальный список недостающих задач** для завершения MVP WorkTime Sync.

Главная цель MVP:

```text
аналитика → задача → ответ сотрудника → решение руководителя/HR → обновление данных
```

Важно: этот документ должен быть понятен сразу трём группам:

1. frontend-команде — какие экраны, кнопки и состояния нужно сделать;
2. backend-команде — какие endpoint’ы, проверки и изменения данных нужны;
3. пользователям платформы — как должна работать логика задач, встреч и подтверждений.

# 2. Общая логика ролей для пользователей платформы

## 2.1. Полный руководитель

Полный руководитель видит всю компанию.

Он может:

- смотреть все отделы;
- смотреть риски по компании;
- видеть задачи по всем отделам;
- назначать задачи любому сотруднику;
- видеть подтверждения графиков;
- контролировать проблемные отделы.

---

## 2.2. HR

HR видит всю компанию, но с HR-фокусом.

HR может:

- видеть HR-расхождения;
- видеть неподтверждённые графики;
- видеть отклонённые графики;
- создавать HR-задачи сотрудникам;
- видеть статусы подтверждений;
- видеть качество данных.

---

## 2.3. Руководитель отдела

Руководитель отдела видит только свой отдел.

Он может:

- смотреть сотрудников своего отдела;
- видеть встречи и конфликты своего отдела;
- создавать задачи сотрудникам своего отдела;
- создавать задачи по встречам сотрудникам своего отдела;
- видеть ответы сотрудников;
- применять решение по переносу встречи;
- закрывать задачи.

Он не может:

- видеть чужие отделы;
- назначать задачи сотрудникам чужого отдела;
- менять задачи сотрудников вне своего отдела.

---

## 2.4. Сотрудник

Сотрудник видит только себя.

Он может:

- открыть личный кабинет;
- посмотреть свой график;
- посмотреть свои встречи;
- посмотреть свои задачи;
- подтвердить задачу;
- отклонить задачу с комментарием;
- предложить другое время для переноса встречи;
- подтвердить свой рабочий график.

Он не может:

- видеть других сотрудников;
- создавать задачи другим;
- менять встречи напрямую;
- финально применять перенос встречи.

# 3. Главная бизнес-логика задач

Задача — это не просто текстовое уведомление.  
Задача — это управленческое действие, которое должно привести к актуализации данных.

Общая схема:

```text
система или руководитель видит проблему
        ↓
создаётся задача
        ↓
сотрудник отвечает
        ↓
руководитель/HR принимает финальное решение
        ↓
данные обновляются или задача закрывается
```

---

# 4. Типы задач

## 4.1. Обычные задачи

| Тип | Назначение |
|---|---|
| `confirm_schedule` | подтвердить рабочий график |
| `review_hr_profile` | проверить HR-профиль |
| `review_load` | проверить нагрузку |
| `update_timezone` | проверить или обновить часовой пояс |

---

## 4.2. Задачи по встречам

| Тип | Назначение |
|---|---|
| `meeting_confirmation` | подтвердить участие во встрече |
| `reschedule_meeting` | согласовать перенос встречи |
| `meeting_outside_work_approval` | согласовать встречу вне рабочего времени |

Важно:

```text
events.csv  → сама встреча
tasks.json  → задача вокруг встречи
```

То есть задача не заменяет встречу.  
Задача только фиксирует действие: подтвердить, перенести, согласовать или отклонить.

---

# 5. Статусы задач

| Статус | Значение |
|---|---|
| `pending` | задача ожидает ответа |
| `confirmed` | сотрудник подтвердил |
| `rejected` | сотрудник отклонил |
| `reschedule_requested` | сотрудник предложил другое время |
| `done` | задача закрыта руководителем/HR |
| `expired` | задача просрочена |

Для статусов:

```text
rejected
reschedule_requested
```

комментарий обязателен.

# 6. Финальная логика задач по встречам

Этот раздел важен для frontend и backend.

## 6.1. Главное правило

Сотрудник **не меняет встречу напрямую**.

Сотрудник только отвечает на задачу:

```text
согласен
не согласен
предлагаю другое время
```

Финально применяет изменение руководитель или HR.

---

## 6.2. Пример: перенос встречи

### Шаг 1. Есть встреча

В `events.csv` есть встреча:

```text
Daily sync
18:30–19:00
```

Она может быть вне рабочего времени сотрудника.

---

### Шаг 2. Руководитель создаёт задачу

Руководитель создаёт задачу:

```text
Тип: reschedule_meeting
Название: Перенести позднюю встречу
Связанная встреча: Daily sync
Текущее время: 18:30–19:00
Предложенное время: 15:00–15:45
```

В данных задачи должны быть поля:

```json
{
  "type": "reschedule_meeting",
  "related_event_id": 12,
  "meeting_action": "reschedule",
  "suggested_start_datetime": "2026-05-19T15:00:00",
  "suggested_end_datetime": "2026-05-19T15:45:00"
}
```

---

### Шаг 3. Сотрудник видит задачу в личном кабинете

Карточка задачи у сотрудника должна выглядеть примерно так:

```text
Задача: Перенести позднюю встречу
Встреча: Daily sync
Текущее время: 18:30–19:00
Предложенное время: 15:00–15:45
Статус: Ожидает ответа
```

---

### Шаг 4. Сотрудник выбирает действие

У сотрудника должны быть кнопки:

```text
[Согласен на перенос]
[Предложить другое время]
[Не согласен]
```

---

## 6.3. Действие сотрудника: “Согласен на перенос”

### Что видит пользователь

Кнопка:

```text
Согласен на перенос
```

### Что отправляет frontend

```json
{
  "status": "confirmed",
  "employee_comment": "Согласен на перенос."
}
```

### Что делает backend

Backend:

1. проверяет, что сотрудник имеет доступ к задаче;
2. меняет статус задачи на `confirmed`;
3. сохраняет комментарий;
4. не меняет `events.csv` сразу.

### Что происходит дальше

После этого руководитель видит кнопку:

```text
[Применить перенос]
```

---

## 6.4. Действие сотрудника: “Предложить другое время”

### Что видит пользователь

Кнопка:

```text
Предложить другое время
```

После нажатия открывается форма:

```text
Новое начало
Новое окончание
Комментарий
```

Пример:

```text
Новое начало: 2026-05-19 16:00
Новое окончание: 2026-05-19 16:45
Комментарий: В 15:00 у меня другая встреча, могу в 16:00.
```

### Что отправляет frontend

```json
{
  "status": "reschedule_requested",
  "employee_comment": "В 15:00 у меня другая встреча, могу в 16:00.",
  "suggested_start_datetime": "2026-05-19T16:00:00",
  "suggested_end_datetime": "2026-05-19T16:45:00"
}
```

### Что делает backend

Backend:

1. проверяет, что комментарий заполнен;
2. проверяет, что передано новое начало и окончание;
3. проверяет, что окончание позже начала;
4. меняет статус задачи на `reschedule_requested`;
5. сохраняет предложенное время;
6. не меняет `events.csv` сразу.

### Что происходит дальше

Руководитель видит:

```text
Сотрудник предложил другое время: 16:00–16:45
Комментарий: В 15:00 у меня другая встреча, могу в 16:00.
```

У руководителя должны быть кнопки:

```text
[Применить предложенный перенос]
[Предложить другое время]
[Закрыть задачу]
```

---

## 6.5. Действие сотрудника: “Не согласен”

### Что видит пользователь

Кнопка:

```text
Не согласен
```

После нажатия открывается поле комментария:

```text
Причина отказа
```

### Что отправляет frontend

```json
{
  "status": "rejected",
  "employee_comment": "Не могу перенести, в это время уже запланирована другая рабочая встреча."
}
```

### Что делает backend

Backend:

1. проверяет, что комментарий заполнен;
2. меняет статус задачи на `rejected`;
3. сохраняет комментарий;
4. не меняет `events.csv`.

### Что происходит дальше

Руководитель видит:

```text
Статус: Отклонено
Причина: Не могу перенести, в это время уже запланирована другая рабочая встреча.
```

У руководителя должны быть кнопки:

```text
[Предложить другое время]
[Создать новую задачу]
[Закрыть задачу]
```

## 6.6. Действие руководителя: “Применить перенос”

### Главное правило

Кнопка “Применить перенос” должна быть доступна только руководителю/HR/executive.

Сотрудник не должен видеть эту кнопку.

### Когда кнопка доступна

Кнопка доступна, если:

```text
task.type = reschedule_meeting
task.status = confirmed или reschedule_requested
task.related_event_id заполнен
task.suggested_start_datetime заполнен
task.suggested_end_datetime заполнен
```

### Как должна выглядеть кнопка

В интерфейсе руководителя:

```text
[Применить перенос]
```

или подробнее:

```text
[Применить перенос встречи]
```

### Что должен вызывать frontend

Нужен backend endpoint:

```text
POST /tasks/{task_id}/apply
```

или:

```text
PATCH /tasks/{task_id}/apply
```

Payload можно оставить пустым:

```json
{}
```

или передавать подтверждение:

```json
{
  "action": "apply_reschedule"
}
```

### Что должен сделать backend

Backend должен:

1. проверить роль пользователя;
2. проверить доступ к задаче;
3. проверить, что задача относится к встрече;
4. проверить, что задача в статусе `confirmed` или `reschedule_requested`;
5. найти событие по `related_event_id`;
6. заменить время события:
   - старое начало → `suggested_start_datetime`;
   - старое окончание → `suggested_end_datetime`;
7. сохранить события через `save_events()`;
8. поменять статус задачи на `done`;
9. добавить запись в history;
10. вернуть обновлённую задачу и обновлённое событие.

### Что видит пользователь после применения

У руководителя:

```text
Перенос применён.
Встреча теперь: 15:00–15:45.
Задача закрыта.
```

У сотрудника:

```text
Статус задачи: Закрыто.
Встреча перенесена на 15:00–15:45.
```

# 7. Логика других задач по встречам

## 7.1. `meeting_confirmation`

Назначение: подтвердить участие во встрече.

### Кнопки у сотрудника

```text
[Подтверждаю участие]
[Не могу участвовать]
```

### Если подтверждает

Frontend отправляет:

```json
{
  "status": "confirmed",
  "employee_comment": "Подтверждаю участие."
}
```

### Если не может участвовать

Frontend отправляет:

```json
{
  "status": "rejected",
  "employee_comment": "Не могу участвовать из-за другой встречи."
}
```

### Применение руководителем

Для `meeting_confirmation` отдельное изменение `events.csv` не обязательно.  
Руководитель может закрыть задачу:

```text
[Закрыть задачу]
```

Backend ставит:

```text
status = done
```

---

## 7.2. `meeting_outside_work_approval`

Назначение: согласовать встречу вне рабочего времени.

### Кнопки у сотрудника

```text
[Согласен на встречу вне рабочего времени]
[Не согласен]
[Предложить перенос]
```

### Если согласен

```json
{
  "status": "confirmed",
  "employee_comment": "Согласен на эту встречу вне рабочего времени."
}
```

### Если не согласен

```json
{
  "status": "rejected",
  "employee_comment": "Не готов участвовать вне рабочего времени."
}
```

### Если предлагает перенос

```json
{
  "status": "reschedule_requested",
  "employee_comment": "Предлагаю перенести на рабочее время.",
  "suggested_start_datetime": "2026-05-19T15:00:00",
  "suggested_end_datetime": "2026-05-19T15:45:00"
}
```

### Применение руководителем

Если сотрудник просто согласился на встречу вне рабочего времени, событие можно не менять.  
Руководитель закрывает задачу:

```text
[Закрыть как согласованную]
```

Если сотрудник предложил перенос, руководитель может:

```text
[Применить перенос]
```

Тогда применяется та же логика, что и для `reschedule_meeting`.

# 9. Что нужно сделать в backend

## 9.1. Добавить применение решения по задаче

Нужен endpoint:

```text
POST /tasks/{task_id}/apply
```

Он нужен, чтобы руководитель/HR/executive мог финально применить решение.

### Для `reschedule_meeting`

Backend должен реально менять время встречи.

### Для `meeting_outside_work_approval`

Если сотрудник предложил перенос, backend должен применить перенос так же, как у `reschedule_meeting`.

### Для `meeting_confirmation`

Backend может просто закрыть задачу как выполненную.

## 9.2. Добавить `save_events()`

save_events(events)

# 14. Финальный сценарий демонстрации

После доработки MVP должен демонстрироваться так:

1. Войти как руководитель отдела.
2. Открыть сотрудника отдела.
3. Увидеть встречу вне рабочего времени.
4. Создать задачу на перенос встречи.
5. Войти как сотрудник.
6. Открыть личный кабинет.
7. Нажать:
   - `Согласен на перенос`,
   - или `Предложить другое время`,
   - или `Не согласен`.
8. Вернуться как руководитель.
9. Увидеть ответ сотрудника.
10. Нажать `Применить перенос`.
11. Система обновляет встречу.
12. Задача закрывается.
13. Конфликт уменьшается или исчезает.

Это и есть завершённый workflow WorkTime Sync.
